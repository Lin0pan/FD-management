## Codebase Patterns
- **Check the working tree and `origin/main` before starting.** This batch began with local `main`
  one commit behind `origin/main` (US-01 had already been squash-merged as #26) and a stale
  unmerged `prd.json` in the index. Branch from `origin/main`, not from local `main`.
- **Parsers throw, they do not return errors.** `parseGroup`, `parseWeekColour` and now
  `parseCardNumber` all raise a typed `DomainError`. Where a screen has to render whether or not the
  rule succeeds, add a *total* form beside the throwing one (`findLowestFreeNumber` /
  `lowestFreeNumber`) and write the throwing one in terms of it — two entry points, one statement of
  the rule. Do not add the total form speculatively; wait for the caller.
- **Domain modules are named for what they do, not for their file**: `formatCardNumber`, not
  `format`. A PRD that writes `format(...)` is naming the concept, not prescribing the export.
- **Derived, never stored**: card numbers, household counts, portion allowance, price. If a value can
  be computed from the birthdates plus the settings in force, it has no column.
- **100% branch coverage on domain + application is a hard gate.** Every guard you add needs a named
  test, so do not add a defensive branch you cannot justify with a rule.
- `eslint`'s `react-hooks/error-boundaries` forbids constructing JSX inside a `try`. Await into a
  variable inside the `try`, build the JSX after it. `npm run build` does not catch this; lint does.
- Next 16 hands a dynamic route its `params` as a **`Promise`** — `await` it, and treat a
  non-numeric id exactly like an unknown one.
- `germanDate()` is duplicated in `einstellungen/page.tsx` and `kunden/[id]/page.tsx`. **Worth
  extracting to a shared UI date formatter in the next story that touches either page** — US-004's
  card view is a candidate.
- Playwright specs share `data/e2e.db` and the customer-number sequence. Assert against the number
  the screen *proposes*, never a hard-coded id, or inserting a spec above breaks the run.
- `prisma migrate dev` refuses to run non-interactively when the dev database is ahead of the
  migrations folder. Delete `data/fd.db` first (`npm run db:reset` does it), then `--create-only`,
  hand-edit the SQL, then `prisma migrate deploy` + `npm run db:seed`.

- **Learnings for future iterations:**
  - **Deviated from the PRD's export names on purpose**, recorded in the story's `notes`: the PRD
    writes `format` / `parse` / `next`, but the file already exported `formatCardNumber` and the
    domain's convention is `parseGroup` / `parseWeekColour`. Read a PRD signature as naming the
    concept, not dictating the identifier; record the deviation rather than silently renaming
    working code.
  - The PRD said `parse` "returns a typed `InvalidCardNumber` error". It **throws** it, because every
    other parser in this domain throws and a lone returning one would make callers branch two ways.
    US-04 (lookup by card number at the counter) is the story that may genuinely want a total
    `tryParseCardNumber` — add it *then*, in terms of this one, not speculatively.
  - The regex is built from the `CARD_INDEX_MARKER` constant with the `i` flag, so the marker is
    stated once. `[1-9][0-9]*` on both parts is what rejects the leading zero and the zero itself in
    one expression — no extra branch, so no extra coverage burden.
  - `next` returns a fresh object; a test asserts the input is untouched. Domain value objects here
    are `readonly` interfaces, never mutated in place.
  - **US-002 next**: the `issueCard` use case. `nextCardNumber` deliberately does *not* decide that a
    card is due — the application layer reads the highest issued index (US-003's `currentCard`) and
    calls it. Validity stays derived from being the highest index; do not add a `valid` column.
