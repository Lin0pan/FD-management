## Codebase Patterns
- Typed domain errors live in `src/domain/errors.ts`: extend the abstract `DomainError` base class,
  declare `readonly code` from the `DomainErrorCode` union, and carry the offending values as
  fields. Never `throw new Error("ŌĆ”")`. Add the new kind to the union when adding a class.
- Domain modules are validated-on-construction: a `createX(input)` factory throws naming the field,
  so no partially-valid object can exist. Prefer this over scattering checks at call sites.
- Coverage on `src/domain` + `src/application` is gated at 100% including **branches**. A
  short-circuiting guard like `Number.isInteger(v) && v >= 1 && v <= 7` needs a test per operand ŌĆö
  budget one test per clause or the coverage gate fails.
- Use a type guard (`function isX(v: number): v is X`) rather than an `as` cast to narrow a
  validated primitive into a literal-union type ŌĆö the repo standard forbids `any` and `!`.
- Test helpers take a `Partial<Input>` overrides object over a valid baseline, so each test names
  exactly the one rule it is about (see `settings.test.ts`).
- Use cases live in `src/application/<area>/<verb-noun>.ts`, one per business action, each taking a
  `Deps` object of ports as its first argument and its `Input` as the second. Ports are declared in
  `src/application/ports.ts` and stay type-only.
- Application tests keep their hand-written fakes **inside the test file** (`settings.test.ts`) ŌĆö a
  separate `fakes.ts` under `src/application/` would be pulled into the 100% coverage gate.
- Every state change appends an `AuditEntry` (`what` / `changedFields` / `when` / `why`, never an
  actor); `when` comes from `clock.now()`, not from any business date in the input.
- A helper like `latestVersion()` that scans with `latest === undefined || ŌĆ”` needs a test where the
  array is out of order, or the second operand stays an uncovered branch.
- Run `npx prettier --write <paths>` before `npm run lint`; the pre-commit hook formats anyway, but
  it rewrites files mid-run otherwise.
- Local quality gate: `npm run lint && npm run typecheck && npm run test:coverage && npm run build`.
- Infrastructure adapters live in `src/infrastructure/prisma/<thing>-repository.ts` and implement a
  port from `src/application/ports.ts`. Rows re-enter the domain through the `createX` factory, so a
  hand-edited database cannot bypass an invariant.
- Integration tests migrate a **throwaway** SQLite file: `mkdtempSync` + `execFileSync("npx",
  ["prisma","migrate","deploy"], { env: { ...process.env, DATABASE_URL: url } })` in `beforeAll`
  (60s timeout), `new PrismaClient({ datasourceUrl: url })`, `rmSync` in `afterAll`. Use an
  **absolute** `file:` path ŌĆö the generated client resolves relative paths against the cwd, while
  the CLI resolves them against `prisma/`.
- SQLite has no enum type: store a union as `String` and narrow it back with a domain parser
  (`parseWeekColour`). Parsing belongs in the domain, not the adapter.
- `prisma migrate dev` needs `DATABASE_URL` on the command line if you do not want it to touch
  `data/fd.db`; delete the scratch db afterwards.
- TypeScript scripts outside `src/` (e.g. `prisma/seed.ts`) run through `tsx`; do not add a
  `package.json#prisma` key for them ŌĆö Prisma 6 warns that it is deprecated.
- A `"use server"` module may export **nothing but async functions** ŌĆö every export becomes a
  callable server endpoint. Put form-state objects and types in a sibling module. `npm run build`
  does **not** catch this; it fails at page load, so a route needs an actual browser/E2E hit.
- A route directory carries its own `deps.ts` composition root ŌĆö the one place real adapters are
  chosen. The page hands that object to a use case and does nothing else with it.
- German text for a rejected field comes from `de.settings.errorFields`, keyed by the `field` string
  the `InvalidSettings` error carries. Add a key when adding a validated field, or the screen quotes
  an English identifier at staff.
- Server components that read data changed through a form need `export const dynamic =
  "force-dynamic"`, or Next prerenders them at build time and the screen shows stale values.
- A screen is a client component only where it genuinely needs state (here: the price grid gaining
  and losing rows). Everything else stays a server component.
- Playwright specs live in `tests/e2e/*.spec.ts`, may import `@/ŌĆ”` (the tsconfig path alias works
  there), and take their expected text from `src/i18n/de.ts` rather than hard-coding German.
- E2E specs that share the seeded `data/e2e.db` must declare `test.describe.configure({ mode:
  "serial" })` ŌĆö `fullyParallel` is on, and two workers writing settings versions race.
- The Playwright `webServer` deletes `data/e2e.db` before migrating and seeding, so every run starts
  from the seed. Don't reintroduce state that assumes leftovers; don't drop the delete either, or a
  same-day second run fails on `RetroactiveSettingsVersion`.
- Address form fields in E2E by their `id` (`#quotaN`, `#reason`, `#effectiveFrom`) ŌĆö a `<label>`
  that also wraps a hint `<span>` gives `getByLabel` an accessible name you did not expect.
