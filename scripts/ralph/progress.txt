## Codebase Patterns
(Carried forward from the US-14 batch, archived at
`scripts/ralph/archive/2026-07-22-us-14-configure-business-rules/progress.txt` â€” read that file for
the full story-by-story history.)

- Typed domain errors live in `src/domain/errors.ts`: extend the abstract `DomainError` base class,
  declare `readonly code` from the `DomainErrorCode` union, and carry the offending values as
  fields. Never `throw new Error("â€¦")`. Add the new kind to the union when adding a class.
- Domain modules are validated-on-construction: a `createX(input)` factory throws naming the field,
  so no partially-valid object can exist. Prefer this over scattering checks at call sites.
- Coverage on `src/domain` + `src/application` is gated at 100% including **branches**. A
  short-circuiting guard needs a test per operand â€” budget one test per clause or the gate fails.
- Use a type guard (`function isX(v: number): v is X`) rather than an `as` cast to narrow a
  validated primitive into a literal-union type â€” the repo standard forbids `any` and `!`.
- Test helpers take a `Partial<Input>` overrides object over a valid baseline, so each test names
  exactly the one rule it is about (see `settings.test.ts`).
- Use cases live in `src/application/<area>/<verb-noun>.ts`, one per business action, each taking a
  `Deps` object of ports as its first argument and its `Input` as the second. Ports are declared in
  `src/application/ports.ts` and stay type-only.
- Application tests keep their hand-written fakes **inside the test file** â€” a separate `fakes.ts`
  under `src/application/` would be pulled into the 100% coverage gate.
- Every state change appends an `AuditEntry` (`what` / `changedFields` / `when` / `why`, never an
  actor); `when` comes from `clock.now()`, not from any business date in the input.
- Run `npx prettier --write <paths>` before `npm run lint`; the pre-commit hook formats anyway, but
  it rewrites files mid-run otherwise.
- Local quality gate: `npm run lint && npm run typecheck && npm run test:coverage && npm run build`.
- Infrastructure adapters live in `src/infrastructure/prisma/<thing>-repository.ts` and implement a
  port from `src/application/ports.ts`. Rows re-enter the domain through the `createX` factory, so a
  hand-edited database cannot bypass an invariant.
- Integration tests migrate a **throwaway** SQLite file: `mkdtempSync` + `execFileSync("npx",
  ["prisma","migrate","deploy"], { env: { ...process.env, DATABASE_URL: url } })` in `beforeAll`
  (60s timeout), `new PrismaClient({ datasourceUrl: url })`, `rmSync` in `afterAll`. Use an
  **absolute** `file:` path â€” the generated client resolves relative paths against the cwd, while
  the CLI resolves them against `prisma/`.
- SQLite has no enum type: store a union as `String` and narrow it back with a domain parser
  (`parseWeekColour`). Parsing belongs in the domain, not the adapter.
- A `"use server"` module may export **nothing but async functions** â€” every export becomes a
  callable server endpoint. Put form-state objects and types in a sibling module. `npm run build`
  does **not** catch this; it fails at page load, so a route needs an actual browser/E2E hit.
- A route directory carries its own `deps.ts` composition root â€” the one place real adapters are
  chosen. The page hands that object to a use case and does nothing else with it.
- German text for a rejected field comes from a keyed map in `src/i18n/de.ts` (e.g.
  `de.settings.errorFields`), keyed by the `field` string the domain error carries.
- Server components that read data changed through a form need `export const dynamic =
  "force-dynamic"`, or Next prerenders them at build time and the screen shows stale values.
- Playwright specs live in `tests/e2e/*.spec.ts`, may import `@/â€¦`, and take their expected text
  from `src/i18n/de.ts` rather than hard-coding German. Specs sharing `data/e2e.db` must declare
  `test.describe.configure({ mode: "serial" })`. Address form fields by `id`, not `getByLabel`.
- **No browser MCP tooling is configured.** For a UI story, verify with a throwaway Playwright run
  against the built app plus a curl of the rendered HTML â€” that is what caught the `"use server"`
  export bug four green gates missed.
- **Faker is a devDependency** as of US-004 (`@faker-js/faker`). Call `faker.seed(â€¦)` once at the top
  of a spec so a failure is reproducible, and keep fixed dates literal â€” only names and addresses
  come from Faker.
- A record that the domain owns is built by a `createX(input, today)` factory in `src/domain/â€¦`; the
  use case above it never validates fields itself. Where a rule already derives something
  (`composition`), call it for its **rejections** and discard the result rather than adding a second
  check â€” one place states the rule.

## [2026-07-22 19:20] - US-001 (HouseholdComposition â€” derive grown-up/children counts)
- Implemented `src/domain/customer/householdComposition.ts`: `composition(members, today)` returning
  `{ grownUps, children }`, plus the `HouseholdMember` / `HouseholdComposition` types and the
  `GROWN_UP_AGE_YEARS = 13` constant. Pure â€” `today` is a parameter, no `new Date()` in the module.
- Added `EmptyHousehold` and `BirthDateInFuture` to `src/domain/errors.ts` and to the
  `DomainErrorCode` union.
- Tests written first (11 named after the rules, not the function): the day before / the day of /
  the day after a 13th birthday, 29 February in a non-leap year on both 28 Feb and 1 March, born
  today, a mixed five-person household, time-of-day independence, empty household, future birthdate,
  and the offending date carried on the error.
- Files changed: `src/domain/customer/householdComposition.ts` (new),
  `src/domain/customer/householdComposition.test.ts` (new), `src/domain/errors.ts`,
  `docs/technical_documentation.md`.
- All gates green: lint, typecheck, coverage (domain + application 100% on all four metrics), build.
- **Learnings for future iterations:**
  - Both dates are normalised to their **UTC calendar day** before comparison, so a birthdate stored
    with a time component (Prisma hands back a `DateTime`) cannot flip a count. Any later date rule
    (card validity, certificate expiry, week colour) should use the same normalisation rather than
    comparing raw timestamps.
  - The 29 February convention falls out of `Date.UTC(year + 13, month, day)` rolling over to
    1 March â€” no special case needed, and thirteen years after a leap year is *never* a leap year,
    so the rollover always happens. Documented in the function's doc comment, per the criterion.
  - `HouseholdMember` is deliberately declared here as *just* `{ birthDate: Date }` â€” the part of a
    member this rule sees. US-004/US-005 will carry names alongside it; a wider record stays
    structurally assignable, so widening the entity need not touch this module. That also kept this
    story free of the Faker dependency (birthdates are not personal data).
  - `src/domain/customer/` was an empty placeholder directory before this story; `card/` and
    `distribution/` still are.

## [2026-07-22 19:22] - US-002 (CustomerNumber â€” lowest free slot allocation)
- Implemented `src/domain/customer/customerNumber.ts`: `lowestFreeNumber(takenNumbers, quotaN)`
  scanning 1..quotaN against a `Set` of taken numbers, returning the first free one.
- Added the `NoFreeCustomerNumber` error class to `src/domain/errors.ts` (the code was already in
  the `DomainErrorCode` union from the initial sketch â€” only the class was missing). It carries
  `quotaN` so the UI can name the limit FD has to raise.
- Tests written first (10, named after the rules): empty list gives 1, `[1,2,4]` gives 3, a
  contiguous range gives the next number, unsorted input, a duplicate taken number, `quotaN = 1`
  free and taken, a full range, numbers above the quota not freeing a slot, and the quota carried
  on the error.
- Files changed: `src/domain/customer/customerNumber.ts` (new),
  `src/domain/customer/customerNumber.test.ts` (new), `src/domain/errors.ts`,
  `docs/technical_documentation.md`.
- All gates green: lint, typecheck, coverage (domain + application 100% on all four metrics), build.
- **Learnings for future iterations:**
  - The PRD wording "returns a typed error *result*" was read as the repo convention of **throwing**
    a typed domain error, matching `composition()` and `createSettings()`. There is no `Result` type
    anywhere in the codebase; introducing one for a single function would have split the error
    handling style in two.
  - Deliberately **no validation** of `quotaN` here. Settings already guarantees an integer â‰Ą 1
    (`requireInteger("quotaN", â€¦)` in `src/domain/policy/settings.ts`), and a second check would be
    an unreachable branch that the 100% branch gate then demands a contrived test for. Where an
    invariant is already enforced on construction, do not re-check it downstream.
  - Ignoring duplicates and out-of-range numbers instead of rejecting them keeps the function total:
    the caller passes whatever the register contains, and the repository query shape can change
    (`DISTINCT` or not) without breaking the rule.
  - `NoFreeCustomerNumber` will be raised in two more places: `registerCustomer` translating the
    partial-unique-index violation (US-004/US-005), and waiting-list promotion (US-12). The
    database, not this function, is the final authority on a free slot.

- Implemented `src/domain/customer/group.ts`: the `Group = "RED" | "BLUE"` type, a `GroupCounts`
  shape, and `suggestGroup({ red, blue })` returning the smaller group, `RED` on a tie.
- Tests written first (5, named after the rules): red bigger, blue bigger, tie, both zero, and a
  one-customer margin. No new error class was needed â€” the function is total.
- Files changed: `src/domain/customer/group.ts` (new), `src/domain/customer/group.test.ts` (new),
  `docs/technical_documentation.md`, `scripts/ralph/prd.json`.
- All gates green: lint, typecheck, coverage (domain + application 100% on all four metrics), build.
- **Learnings for future iterations:**
  - `Group` is deliberately a **separate type** from `WeekColour` in `src/domain/policy/settings.ts`
    despite both being `"RED" | "BLUE"`. A week's colour is derived from the settings anchor; a
    customer's group is stored and hand-editable. Aliasing them would let one be changed through the
    other. US-005 will therefore need its own `parseGroup` for the persisted string â€” do not reuse
    `parseWeekColour`, whose `InvalidSettings` error names a settings field.
  - `counts.blue < counts.red ? "BLUE" : "RED"` gets the tie rule and both branches from a single
    comparison, so the branch gate is satisfied by the four ordering tests without a contrived case.
    Writing it as two `if`s would have added an unreachable third branch.
  - The suggestion is advice only â€” US-004 must let an explicit group in the input win over it, and
    must not re-derive the group at persistence time.
