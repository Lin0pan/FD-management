## Codebase Patterns
- Domain errors: add a `code` to the `DomainErrorCode` union in `src/domain/errors.ts` AND a class
  whose `readonly code` matches (convention is code === className); carry the values that made it
  fail so the UI can render German without re-deriving. Throw with `throw new X(...)`, never
  `throw new Error(...)`.
- Calendar-day comparisons come in TWO flavours. Week colour and distribution-day rules compare the
  **UTC** day via `startOfUtcDay` (weekColour.ts) â€” a configured week where the minute is irrelevant.
  **Attendance** (once-per-day, same-day correction) compares the **Europe/Berlin** wall-clock day via
  `Intl.DateTimeFormat("en-CA", { timeZone: "Europe/Berlin", ... }).format()` giving a `YYYY-MM-DD`
  key. `Intl` is pure and allowed in the domain (no wall-clock read, no I/O). Pick the right one.
- Time is a parameter in domain/application (`today: Date`, `deps.clock.now()`); a zero-arg
  `new Date()` is an ESLint error there. `new Date(someValue)` is fine.
- Domain modules return discriminated unions or typed sentinels for verdicts (see counterVerdict's
  `Verdict`, attendance's `"OK" | AlreadyServedToday`), rather than throwing, when the caller must
  branch on the outcome. The application layer decides whether to throw.
- Coverage on domain/ and application/ is gated at 100% â€” every branch of a new pure function needs a
  named test. UI/infra are not gated.
- German UI strings live only in `src/i18n/de.ts`. The counter already has
  `ausgabe.verdicts.alreadyServedToday` (headline "Heute bereits ausgegeben").
- Prisma adapters live in `src/infrastructure/prisma/<name>-repository.ts` and implement a port from
  `application/ports.ts`. Map a `P2002` unique-constraint violation to a domain error by
  `JSON.stringify(error.meta ?? {}).includes("<column>")` (see card/customer/distribution repos); map
  `P2025` (record-not-found on update/delete) to the not-found domain error. Integration tests migrate
  a throwaway SQLite file in `beforeAll` (`prisma migrate deploy` with a tmp `DATABASE_URL`), delete
  rows in `beforeEach`, and use Faker with a fixed seed. Infra is NOT coverage-gated.
- **The `init` migration is regenerated, never amended** (pre-release, history disposable). To add a
  table: edit schema.prisma, `rm -rf` the old `prisma/migrations/*_init`, run
  `npx prisma migrate dev --name init --create-only`, then **re-add the hand-written partial unique
  slot index** (`Customer_customerNumber_onRegister_key`) at the end of the new migration.sql â€” Prisma
  cannot express it and regenerating drops it. Finish with `npm run db:reset`.
- **DATABASE_URL is `file:../data/fd.db` in `.env`** (relative to `prisma/`, resolving to repo-root
  `data/fd.db`). Do NOT override it with `file:./data/fd.db` on the command line â€” that resolves to
  `prisma/data/fd.db`, a stray untracked DB the `.gitignore` (`/data/*.db`) does not catch. Let the
  `.env` value stand; only tests pass a tmp URL.
- **Server-action forms on the counter (US-05.4 pattern).** A `"use server"` `actions.ts` beside the
  page holds the write actions; a plain non-server module (`serve-state.ts`) holds the useActionState
  state types (a `"use server"` module may export only async functions). A `<input type="checkbox">`
  submits nothing when unchecked, so a boolean flag is read as **presence**:
  `formData.get("paid") !== null` â€” and the box is pre-checked in markup to match the default. Split
  the composition root into a read deps object (no audit log) and a write deps object (`+ audit`) so
  the page physically cannot write. A native `<details>`/second-submit-button gives a keyboard-only
  confirmation step with no client JS. After a write, `revalidatePath` re-renders the page and the
  client component may switch its rendered branch, but its `useActionState` state persists (same
  instance) â€” rely on that to keep a confirmation + a focus `useEffect` firing across the switch.
- **`next build` touches `next-env.d.ts`** â€” `git checkout next-env.d.ts` before committing so the
  generated file does not land in the diff.
- **E2E specs share one `data/e2e.db` and own disjoint customer-number ranges.** A spec that seeds
  rows through Prisma must pick numbers no other spec uses (counter: 201-206 + 239; portions: 211;
  serve: 221-222) â€” grep `tests/e2e/*.spec.ts` first. `customerNumber` is NOT a Prisma `findUnique`
  key (uniqueness is a hand-written partial index), so read seeded rows back with `findFirst`. The
  clock is pinned by writing an ISO instant to `data/e2e-now.txt` (`FD_FIXED_NOW_FILE`); delete it in
  `afterAll` or the settings specs freeze. Off-CI, `reuseExistingServer` reuses a running server
  WITHOUT resetting the db (reset is in the webServer boot command) â€” free port 3000 or use `CI=1`
  for a clean boot, and run `--retries=0` when debugging, because a retry re-runs `beforeAll` and
  surfaces the real failure as a bogus unique-customerNumber seed collision.

- Files changed: tests/e2e/serve.spec.ts (new), docs/technical_documentation.md (file tree + E2E

- **Learnings for future iterations:**
  - **E2E customer-number ownership is a shared-db contract.** Every spec that inserts rows straight
    through Prisma owns a number range in the shared `data/e2e.db`, and they MUST NOT overlap: counter
    owns 201-206 + 239, portions owns **211**, serve now owns 221-222. Picking 211/212 first silently
    collided with portions.spec â€” grep `tests/e2e/*.spec.ts` for your intended numbers before seeding.
  - **A retry masks the real E2E failure as a seed collision.** With CI retries=2, a failing test
    re-runs `beforeAll`, whose `prisma.customer.create` then hits the unique customerNumber from the
    first attempt â€” so the surfaced error is "Unique constraint failed on customerNumber" at the seed
    line, not the actual assertion. Re-run with `--retries=0` (and a fresh server) to see the true
    first failure.
  - **`customerNumber` is not a Prisma `findUnique` key** â€” its uniqueness is a hand-written partial
    index Prisma's schema cannot see. Read a seeded row back with `findFirst({ where: { customerNumber } })`.
  - **Leftover `next start` servers break local re-runs.** `reuseExistingServer` is true off-CI, so a
    server left from a prior run is reused *without* resetting the db (the reset is in the webServer
    boot command). Free port 3000 (`for pid in $(lsof -ti:3000); do kill -9 $pid; done`) before a
    clean local run, or use `CI=1` to force a fresh reset boot. `pkill -f next` exited 144 in this
    sandbox â€” kill by pid from `lsof -ti:3000` instead.
  - The confirmation/record time is Berlin wall-clock: 09:00Z on 08.01.2026 renders as 10:00 (UTC+1
    in January). Compute it with the app's own `germanTime(new Date(TODAY))` rather than hard-coding.
