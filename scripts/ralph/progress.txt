## Codebase Patterns
- Domain modules are pure and take dates as parameters — `new Date()` with no argument is an ESLint
  error in `src/domain` and `src/application`. `new Date(value)` is fine.
- Calendar rules work on **UTC day instants** (`Date.UTC(y, m, d)`), never on raw timestamps or local
  time — see `householdComposition.ts` and `distribution/weekColour.ts` for the idiom.
- Coverage on `src/domain` and `src/application` is gated at **100% including branches**. Every
  branch of a helper needs a named test, so do not add a defensive branch you cannot reach.
- Typed errors from `src/domain/errors.ts` only — add a code to `DomainErrorCode` before the class.
- Test names state the business rule, not the function: "flips on the Sunday-to-Monday boundary".
- `docs/technical_documentation.md` carries both a source tree and a per-module section; a new domain
  module needs an entry in each.
- Full gate: `npm run lint && npm run typecheck && npm run test:coverage && npm run build`.
- Shared calendar arithmetic lives in `distribution/weekColour.ts` (`startOfUtcDay`, `isoWeekdayOf`) —
  import it rather than re-deriving UTC/ISO weekday maths in a new module.
- Verify every ISO date claim in a test comment against a real calendar before trusting it; an
  expectation written from intuition is the likeliest red in this area.

## 2026-07-23 - US-001 (weekColour calendar alternation rule)
- Implemented `src/domain/distribution/weekColour.ts`: `colourOf(date, anchor)` derives the RED/BLUE
  week colour by ISO-week parity from the configured anchor, and `isoWeekOf(date)` formats the ISO
  week as `2026-W30` for the lookup UI (US-004 will need it).
- Written test-first: the spec was red (module missing) before the implementation existed.
- Files changed:
  - `src/domain/distribution/weekColour.ts` (new)
  - `src/domain/distribution/weekColour.test.ts` (new, 23 tests)
  - `docs/technical_documentation.md` (source tree + a module section)
  - `scripts/ralph/prd.json` (US-001 passes)
- Gate: lint, typecheck, `test:coverage` (310 tests, 100% on domain + application) and `build` all
  green.
- **Learnings for future iterations:**
  - ISO facts verified against a scratch script rather than intuition, as the PRD demands:
    2026 is a 53-week ISO year (1 Jan 2026 is a Thursday); 1 Jan 2023 falls in `2022-W52`;
    1 Jan 2027 falls in `2026-W53`; `2026-W02` starts Monday 5 January 2026.
  - The ISO year of a week is the calendar year of its **Thursday** — that single rule makes both the
    December and the January crossovers fall out; and 28 December is always in the last week of its
    ISO year, which is how `weeksInIsoYear` is computed.
  - Negative parity needs `((n % 2) + 2) % 2`; a plain `n % 2` is negative in JS and would make every
    pre-anchor week report the anchor colour.
  - `createSettings` validates only the *shape* of `weekAnchor.isoWeek` (a regex allowing W01–W53).
    `2025-W53` therefore passes settings validation but names no real week, so `weekColour.ts`
    re-checks existence and raises `InvalidSettings` against the same field. **Open follow-up:** the
    settings form (US-14) can still save such an anchor, and it would only fail when `/ausgabe`
    renders. Worth tightening in `createSettings` when US-014 is revisited — it was left alone here
    to keep this story to one intent.
  - `tsconfig.json` does **not** set `noUncheckedIndexedAccess`, so `RegExpExecArray[1]` is `string`
    and needs no guard — a guard would be an uncovered branch and fail the coverage gate.
  - There is no `e2e/` directory yet; `npm run test:e2e` runs Playwright from `tests/e2e` per
    `vitest.config.ts`'s comment. US-005 will have to create it.

## 2026-07-23 - US-002 (isDistributionDay and nextDistribution)
- Implemented `src/domain/distribution/distributionDay.ts`: `isDistributionDay(date, weekday)` and
  `nextDistribution(date, settings)` returning `{ date, colour }` for the next distribution at or

- Written test-first: the spec was red (module missing) before the implementation existed.
- `weekColour.ts` grew two exports, `startOfUtcDay` and `isoWeekdayOf`, replacing its two private
  helpers so the new module reuses the same UTC/ISO arithmetic instead of duplicating it.
- Files changed:
  - `src/domain/distribution/distributionDay.ts` (new)
  - `src/domain/distribution/distributionDay.test.ts` (new, 13 tests)
  - `src/domain/distribution/weekColour.ts` (helpers exported)
  - `docs/technical_documentation.md` (source tree + a module section)
  - `scripts/ralph/prd.json` (US-002 passes)
- Gate: lint, typecheck, `test:coverage` (323 tests, 100% on domain + application) and `build` all
  green.
- **Learnings for future iterations:**
  - `date.getUTCDay()` is unaffected by the time of day, so the ISO weekday needs no day-normalising
    first — only the returned *date* does.
  - "Next occurrence of weekday W, today counting" is `(W - isoWeekday(d) + 7) % 7` days ahead; the
    `% 7` is what makes a distribution day return itself rather than +7.
  - One test failed first time because its comment mis-stated the calendar: 2025-W52 (22–28 Dec) is
    **two** weeks before the anchor 2026-W02, not one — 2025-W53 does not exist, so W52 is followed
    by 2026-W01. Check the week numbering, not just the day count.
  - `Settings.distributionWeekday` is already narrowed to `IsoWeekday` by `createSettings`, so the
    domain functions can take the narrow type and need no re-validation branch (which would be an
    uncovered branch under the 100% gate).
  - Next up (US-003) needs only `SettingsRepository` + `Clock`; the fake-based application tests in
    `src/application/settings/settings.test.ts` are the pattern to copy.
