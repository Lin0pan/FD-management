## Codebase Patterns
- Domain modules are pure: `today` (and `weekColour`, settings) are always parameters ‚Äî never
  `new Date()`. Day-boundary comparisons go through `startOfUtcDay` (exported from
  `domain/distribution/weekColour.ts`) so a rule never turns on the time of day.
- Domain tests use relative imports (`./counterVerdict`); application/infra use the `@/` alias.
- Domain/application coverage is gated at 100% ‚Äî avoid `?? ""`-style fallback branches you cannot
  exercise; pass values straight through (e.g. `reason: string | null`) instead.
- Test helper convention: an `on("2026-07-23")` ‚Üí UTC-midnight `Date`, and a `customer(overrides)`
  factory that returns the happy-path value so each test overrides only the field its rule turns on.
- Verdicts/statuses/reasons are discriminated unions with a `kind`/enum discriminator; the UI switch
  is meant to be exhaustive with a `never`-check (US-04.4).
- Infra tests migrate a throwaway SQLite file in `beforeAll` (`mkdtempSync` + `prisma migrate
  deploy`) and `deleteMany()` in `beforeEach`; keep the temp `url` at module scope if a test needs a
  second client. Cascades mean deleting customers clears households/cards/certificates too.
- Prisma's SQLite provider has **no join strategy**, so `include` costs one statement per relation.
  Assert query counts are *equal across input sizes* (no fan-out), never a magic number ‚Äî a new
  relation would break the latter for no reason.
- `insertCustomer` in the repository test writes a bare row with no certificate or card: fine for
  counting/slot tests, but any read through `toRegisteredCustomer` throws `InvalidCustomerRecord`.
  Use `repository.create(newCustomer({...}))` for anything that reads a customer back.

    (NOT_FOUND, ARCHIVED, BLOCKED, WRONG_GROUP, OUTDATED_CARD, ALREADY_SERVED_TODAY, CLEAR_TO_SERVE,
    CLEAR_TO_SERVE_CERTIFICATE_EXPIRED), `CounterCustomer`/`CounterInput` inputs, and the function.

    and certificate-boundary tests (day before/of/after + 29 Feb).
  - `docs/technical_documentation.md` ‚Äî added the module to the file tree.
- **Learnings for future iterations:**
  - Precedence is fixed: NOT_FOUND ‚Üí ARCHIVED ‚Üí BLOCKED ‚Üí WRONG_GROUP ‚Üí OUTDATED_CARD ‚Üí certificate
    check ‚Üí CLEAR_TO_SERVE. Status values (ACTIVE/BLOCKED/ARCHIVED) are mutually exclusive, so the
    "adjacent pair" tests for the status verdicts show the higher status winning over a *lower*
    condition (e.g. an archived/blocked customer also in the wrong group) rather than over each other.
  - `ALREADY_SERVED_TODAY` is declared in the union but the read-only rule never emits it ‚Äî US-05's
    serve/duplicate-prevention will. It exists now so US-004's exhaustive UI switch already covers it.
  - `OUTDATED_CARD` only fires for a *presented* index strictly `< currentCardIndex`; a bare customer
    number (`presentedCardIndex: null`) always means the current card. `currentCardIndex` is the
    highest issued card index ‚Äî the application layer derives it from the card run (US-04.2/US-04.3).
  - `BLOCKED.reason` / `CounterCustomer.blockReason` are `string | null` (US-08 not built yet; the
    use case will pass `null` until blocks exist). Kept nullable to avoid an unexercisable fallback.
  - Certificate expiry is a calendar-day comparison via `startOfUtcDay`; valid *through* validUntil.

## 2026-07-23 - US-003 (Counter lookup queries and indexes)
- Proved the counter's persistence read (US-04.3). The implementation had already landed with
  US-002 ‚Äî `findByCustomerNumber`, `CUSTOMER_INCLUDE` and both indexes existed ‚Äî so this story was
  the integration tests plus one corrected claim, and **no migration** (the schema was already right).
- Files changed:
  - `src/infrastructure/prisma/customer-repository.test.ts` ‚Äî six specs for `findByCustomerNumber`
    (reassigned slot resolves to the active holder; a blocked holder still holds the slot; the most
    recently archived holder when the slot stands empty; null for an unheld number; relations read
    back whole; and a no-fan-out spec) plus one asserting the two counter indexes exist via
    `PRAGMA index_list("Customer")`.
  - `src/infrastructure/prisma/customer-repository.ts` ‚Äî corrected the `CUSTOMER_INCLUDE` comment,
    which claimed a "single query".
  - `docs/technical_documentation.md` ‚Äî documented why `findByCustomerNumber` is two queries and
    what the N+1 guarantee actually is.
- Checks: lint, typecheck, `test:coverage` (390 tests, domain+application still 100%) and build all
  green. No UI change, so no browser verification was needed.
- **Learnings for future iterations:**
  - The PRD's "loaded in one query" is not achievable on SQLite: `relationLoadStrategy: "join"` is
    Postgres/MySQL only, so Prisma issues the customer plus one SELECT per relation ‚Äî four for a
    counter lookup. I measured it with a throwaway `PrismaClient({ log: [{ emit: "event", level:
    "query" }] })` before writing the assertion. The real invariant is that the count is constant in
    household size, so the test compares a 7-person household against a 2-person one. Recorded the
    deviation in the story's `notes` rather than quietly claiming the criterion.
  - A query listener cannot be detached (`$on` has no `$off` in this Prisma version), so a test that
    counts queries needs a fresh client per measurement, disconnected afterwards.
  - `findByCustomerNumber` is deliberately two round trips, not one `orderBy`: an active holder must
    beat an archived one regardless of row age, and recency only decides *among* archived holders.
    A single ordered query cannot say that.
  - US-004 (next) is the first UI story of this batch: `/ausgabe` already exists as a route, and the
    exhaustive `never`-check switch over the `Verdict` union is an explicit criterion. All eight
    verdict cases need German strings in `src/i18n/de.ts`, `ALREADY_SERVED_TODAY` included even
    though the read-only rule never emits it yet.
