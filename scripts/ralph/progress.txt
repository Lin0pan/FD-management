## Codebase Patterns
- Domain errors: add a `code` to the `DomainErrorCode` union in `src/domain/errors.ts` AND a class
  whose `readonly code` matches (convention is code === className); carry the values that made it
  fail so the UI can render German without re-deriving. Throw with `throw new X(...)`, never
  `throw new Error(...)`.
- Calendar-day comparisons come in TWO flavours. Week colour and distribution-day rules compare the
  **UTC** day via `startOfUtcDay` (weekColour.ts) â€” a configured week where the minute is irrelevant.
  **Attendance** (once-per-day, same-day correction) compares the **Europe/Berlin** wall-clock day via
  `Intl.DateTimeFormat("en-CA", { timeZone: "Europe/Berlin", ... }).format()` giving a `YYYY-MM-DD`
  key. `Intl` is pure and allowed in the domain (no wall-clock read, no I/O). Pick the right one.
- Time is a parameter in domain/application (`today: Date`, `deps.clock.now()`); a zero-arg
  `new Date()` is an ESLint error there. `new Date(someValue)` is fine.
- Domain modules return discriminated unions or typed sentinels for verdicts (see counterVerdict's
  `Verdict`, attendance's `"OK" | AlreadyServedToday`), rather than throwing, when the caller must
  branch on the outcome. The application layer decides whether to throw.
- Coverage on domain/ and application/ is gated at 100% â€” every branch of a new pure function needs a
  named test. UI/infra are not gated.
- German UI strings live only in `src/i18n/de.ts`. The counter already has
  `ausgabe.verdicts.alreadyServedToday` (headline "Heute bereits ausgegeben").
