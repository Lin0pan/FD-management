## Codebase Patterns
- Typed domain errors live in `src/domain/errors.ts`: extend the abstract `DomainError` base class,
  declare `readonly code` from the `DomainErrorCode` union, and carry the offending values as
  fields. Never `throw new Error("…")`. Add the new kind to the union when adding a class.
- Domain modules are validated-on-construction: a `createX(input)` factory throws naming the field,
  so no partially-valid object can exist. Prefer this over scattering checks at call sites.
- Coverage on `src/domain` + `src/application` is gated at 100% including **branches**. A
  short-circuiting guard like `Number.isInteger(v) && v >= 1 && v <= 7` needs a test per operand —
  budget one test per clause or the coverage gate fails.
- Use a type guard (`function isX(v: number): v is X`) rather than an `as` cast to narrow a
  validated primitive into a literal-union type — the repo standard forbids `any` and `!`.
- Test helpers take a `Partial<Input>` overrides object over a valid baseline, so each test names
  exactly the one rule it is about (see `settings.test.ts`).
- Use cases live in `src/application/<area>/<verb-noun>.ts`, one per business action, each taking a
  `Deps` object of ports as its first argument and its `Input` as the second. Ports are declared in
  `src/application/ports.ts` and stay type-only.
- Application tests keep their hand-written fakes **inside the test file** (`settings.test.ts`) — a
  separate `fakes.ts` under `src/application/` would be pulled into the 100% coverage gate.
- Every state change appends an `AuditEntry` (`what` / `changedFields` / `when` / `why`, never an
  actor); `when` comes from `clock.now()`, not from any business date in the input.
- A helper like `latestVersion()` that scans with `latest === undefined || …` needs a test where the
  array is out of order, or the second operand stays an uncovered branch.
- Run `npx prettier --write <paths>` before `npm run lint`; the pre-commit hook formats anyway, but
  it rewrites files mid-run otherwise.
- Local quality gate: `npm run lint && npm run typecheck && npm run test:coverage && npm run build`.
- Infrastructure adapters live in `src/infrastructure/prisma/<thing>-repository.ts` and implement a
  port from `src/application/ports.ts`. Rows re-enter the domain through the `createX` factory, so a
  hand-edited database cannot bypass an invariant.
- Integration tests migrate a **throwaway** SQLite file: `mkdtempSync` + `execFileSync("npx",
  ["prisma","migrate","deploy"], { env: { ...process.env, DATABASE_URL: url } })` in `beforeAll`
  (60s timeout), `new PrismaClient({ datasourceUrl: url })`, `rmSync` in `afterAll`. Use an
  **absolute** `file:` path — the generated client resolves relative paths against the cwd, while
  the CLI resolves them against `prisma/`.
- SQLite has no enum type: store a union as `String` and narrow it back with a domain parser
  (`parseWeekColour`). Parsing belongs in the domain, not the adapter.
- `prisma migrate dev` needs `DATABASE_URL` on the command line if you do not want it to touch
  `data/fd.db`; delete the scratch db afterwards.
- TypeScript scripts outside `src/` (e.g. `prisma/seed.ts`) run through `tsx`; do not add a
  `package.json#prisma` key for them — Prisma 6 warns that it is deprecated.
- A `"use server"` module may export **nothing but async functions** — every export becomes a
  callable server endpoint. Put form-state objects and types in a sibling module. `npm run build`
  does **not** catch this; it fails at page load, so a route needs an actual browser/E2E hit.
- A route directory carries its own `deps.ts` composition root — the one place real adapters are
  chosen. The page hands that object to a use case and does nothing else with it.
- German text for a rejected field comes from `de.settings.errorFields`, keyed by the `field` string
  the `InvalidSettings` error carries. Add a key when adding a validated field, or the screen quotes
  an English identifier at staff.
- Server components that read data changed through a form need `export const dynamic =
  "force-dynamic"`, or Next prerenders them at build time and the screen shows stale values.
- A screen is a client component only where it genuinely needs state (here: the price grid gaining
  and losing rows). Everything else stays a server component.

  `DomainError` base + `InvalidSettings`, `NoSettingsInForce`, `NoPriceForHousehold`.

  30 tests, written first), `src/domain/errors.ts`, `docs/technical_documentation.md`.
- All gates green; domain coverage 100% on statements/branches/functions/lines.
- **Learnings for future iterations:**
  - `resolveSettingsAt` does not assume the versions array is sorted — it scans for the greatest
    `effectiveFrom <= date`. Repository adapters therefore need not order their query.
  - `SettingsInput` widens only `distributionWeekday` to `number`; everything else reuses
    `Settings`, so adding a policy field means touching only the `Settings` interface plus its
    validation and the returned object.
  - Boundary semantics settled here: a version dated exactly `date` **is** in force that day, and a
    date before the earliest version throws rather than returning a default.
  - `priceFor` matches exactly and never interpolates — US-14.3's seed must therefore enumerate
    every household size FD actually has, not just a formula.
  - No Faker in `package.json` yet; the standard asks for synthetic data via Faker, so a story
    needing names/addresses will have to add it (settings tests need only numbers).

## [2026-07-22] - US-002 (SettingsRepository port + readCurrentSettings / updateSettings)
- Added the `SettingsRepository`, `CustomerCounter` and `AuditLog` ports (plus the `AuditEntry`
  shape) to `src/application/ports.ts`, alongside the existing `Clock`.
- Implemented `readCurrentSettings(deps)` (resolves versions against `deps.clock.now()`) and
  `updateSettings(deps, input)` — validates values, requires a reason, refuses a version dated on or
  before the latest one, refuses a quota below the active customer count, appends a new version and
  writes the audit entry. Nothing is written unless every check passes.
- Added `changedSettingsFields(previous, next)` to the domain settings module so the audit entry can
  name *what* changed; added `QuotaBelowActiveCustomers`, `RetroactiveSettingsVersion` and
  `MissingAuditReason` to `src/domain/errors.ts`.
- Files changed: `src/application/ports.ts`, `src/application/settings/read-current-settings.ts`
  (new), `src/application/settings/update-settings.ts` (new),
  `src/application/settings/settings.test.ts` (new, 22 tests, written first),
  `src/domain/policy/settings.ts`, `src/domain/policy/settings.test.ts`, `src/domain/errors.ts`,
  `docs/technical_documentation.md`.
- All gates green (lint, typecheck, coverage 100% on domain + application, build).
- **Learnings for future iterations:**
  - Equal `effectiveFrom` dates are **rejected**, not just earlier ones — US-14.3 puts a unique
    constraint on the column, so a same-day re-save must pick the next day (or the UI must offer to
    supersede). The error carries both dates.
  - The audit diff compares against the **latest** version, not the one in force today, because that
    is the version the new one supersedes.
  - `updateSettings` returns the stored `Settings`, which saves the UI a re-read after saving.
  - `ports.ts` reports 0% coverage in the v8 table because it transpiles to nothing; that is
    harmless (0/0) and does not affect the global threshold.
  - Still no Faker dependency; nothing in this story needed names or addresses.
