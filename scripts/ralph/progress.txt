## Codebase Patterns
- Typed domain errors live in `src/domain/errors.ts`: extend the abstract `DomainError` base class,
  declare `readonly code` from the `DomainErrorCode` union, and carry the offending values as
  fields. Never `throw new Error("â€¦")`. Add the new kind to the union when adding a class.
- Domain modules are validated-on-construction: a `createX(input)` factory throws naming the field,
  so no partially-valid object can exist. Prefer this over scattering checks at call sites.
- Coverage on `src/domain` + `src/application` is gated at 100% including **branches**. A
  short-circuiting guard like `Number.isInteger(v) && v >= 1 && v <= 7` needs a test per operand â€”
  budget one test per clause or the coverage gate fails.
- Use a type guard (`function isX(v: number): v is X`) rather than an `as` cast to narrow a
  validated primitive into a literal-union type â€” the repo standard forbids `any` and `!`.
- Test helpers take a `Partial<Input>` overrides object over a valid baseline, so each test names
  exactly the one rule it is about (see `settings.test.ts`).
- Use cases live in `src/application/<area>/<verb-noun>.ts`, one per business action, each taking a
  `Deps` object of ports as its first argument and its `Input` as the second. Ports are declared in
  `src/application/ports.ts` and stay type-only.
- Application tests keep their hand-written fakes **inside the test file** (`settings.test.ts`) â€” a
  separate `fakes.ts` under `src/application/` would be pulled into the 100% coverage gate.
- Every state change appends an `AuditEntry` (`what` / `changedFields` / `when` / `why`, never an
  actor); `when` comes from `clock.now()`, not from any business date in the input.
- A helper like `latestVersion()` that scans with `latest === undefined || â€¦` needs a test where the
  array is out of order, or the second operand stays an uncovered branch.
- Run `npx prettier --write <paths>` before `npm run lint`; the pre-commit hook formats anyway, but
  it rewrites files mid-run otherwise.
- Local quality gate: `npm run lint && npm run typecheck && npm run test:coverage && npm run build`.

- **Learnings for future iterations:**
  - `resolveSettingsAt` does not assume the versions array is sorted â€” it scans for the greatest
    `effectiveFrom <= date`. Repository adapters therefore need not order their query.
  - `SettingsInput` widens only `distributionWeekday` to `number`; everything else reuses
    `Settings`, so adding a policy field means touching only the `Settings` interface plus its
    validation and the returned object.
  - Boundary semantics settled here: a version dated exactly `date` **is** in force that day, and a
    date before the earliest version throws rather than returning a default.
  - `priceFor` matches exactly and never interpolates â€” US-14.3's seed must therefore enumerate
    every household size FD actually has, not just a formula.
  - No Faker in `package.json` yet; the standard asks for synthetic data via Faker, so a story
    needing names/addresses will have to add it (settings tests need only numbers).

- **Learnings for future iterations:**
